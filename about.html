What is powerdelivery?

A free, open-source toolkit for automating deployment of software features using technologies that are easier to work with by IT operations personnel and enable higher quality and more frequent releases by following Continuous Delivery principles.

Continuous Delivery is a term coined in the principles of the Agile Manifesto:

"Our highest priority is to satisfy the customer through early and continuous delivery of valuable software."

Continuous delivery is also the name of a book by Jez Humble of Thoughtworks and David Farley.

What does it do?

The philosophy behind Continuous Delivery is that teams need to release more often to their customers to get feedback on their ideas before investing so deeply in them. This helps teams get to the truly valuable features faster and also eliminates waste by identifying bad ideas early. To do this effectively requires coaching in the approach to really succeed but typically boils down to these essential practices:

  1. Use SCRUM to manage the workload of your team.
  2. Define requirements using test scripts called acceptance criteria.
  3. Use automated testing to reduce cycle time, or the time between the conception of an idea when and it is available to users.
  4. Deploy into your development, test, and production environments with automation and never let people modify your environment directly.

powerdelivery supports all of these practices, but does the heavy lifting to enable primarily the last one.

What is required to use it?

Currently powerdelivery is designed to work with Microsoft Team Foundation Server (TFS). TFS is an Application Lifecycle Management (ALM) tool that provides many great features that can be leveraged by teams starting to do continuous delivery. However, powerdelivery only relies on the Team Build feature, and should you choose to use a different project management tool, you can still use it to get the same benefits for deployment automation.

How is it different?

More than simply "yet another automated deployment technology", powerdelivery was designed specifically to follow Continuous Delivery practices making your team more efficient by eliminating the possibility of releasing untested changes to production. It is also designed to be easier to work with than the existing tools from Microsoft for automated deployment by themselves. Microsoft recently acquired Incycle Software's In Release product which is meant to provide a similar deployment solution as powerdelivery, however it requires understanding and use of Windows Workflow Foundation which has proven to be difficult for IT operations personnel to learn. It also encourages teams to think of the release to their environments as separately designed processes.

The core of powerdelivery is a simple framework built on top of Windows PowerShell. Everything you can do with the massive array of existing PowerShell scripts and modules available from Microsoft and on the web is available to you from powerdelivery. And once you write the deployment, you can target each of your environments using the same script.

Why Windows PowerShell?

TFS is great at watching your source code repository for submissions and kicking off builds, but unless the features that come with the build out of the box are enough for you, it requires significant customization to be suitable for a Continuous Delivery build platform. I highly recommend you read Jez Humble's book he makes a much more detailed case about why it makes sense to author your builds in a scripting language. This information will come in handy as you run across folks who may be confused as to why you wouldn't just do it all within TFS build itself and need to educate them.

I'll summarize what's in the book to highlight the most important benefits. First, if your automated build relies wholly on TFS, you can't run a build on your computer that does everything that the server can. Second, if you want to really tap into automation you will be modifying the build's behavior greatly and that requires recompilation of code in the case of MSBuild, or understanding the difficult Windows Workflow engine that TFS leverages. Finally, to really automate as much as possible you will be setting up and tearing down software and hardware infrastructure needed by your environments as well as doing typical build activities (compiling, database changes) and most IT operations personnel are more familiar with PowerShell than C# code or Windows Workflow foundation as it is similar in use to bash shell scripting and batch files.

Remember, continuous delivery means continuously changing your release process as well.

What is included?

  • A simple PowerShell framework for deploying your software into multiple environments.
  • A "hook" into Microsoft Team Foundation Server that triggers your script when you "kickoff" a build.
  • A mechanism for storing environment configuration settings in .yml or .csv files. For instance, your local computer, development, test, and production environment probably use a different database.
  • PowerShell cmdlets (functions) and modules (libraries) that help with writing builds.
  • PowerShell scripts that can be used to setup security appropriately between your automated build server and the computers it deploys to.
